# Tempo vs. Tiwari et al. (MSR 2025): 151-Bug Architectural Review

This document evaluates each of the 151 date/time bugs catalogued by Tiwari, Chen, Joukov, Vandervelde, Li and Padhye in *It's About Time: An Empirical Study of Date and Time Bugs in Open-Source Python Software* (MSR 2025, Distinguished Paper Award) against Tempo's architectural design. For each bug we record whether the bug class is **in scope** for Tempo (i.e. the kind of failure Tempo's design aims to prevent or handle) and, when in scope, whether Tempo **avoids it by design**, **addresses** it through explicit machinery, **partially** handles it, or has a **gap**.

## Method and caveats

Tiwari et al.'s analysis classifies each bug along three axes: a conceptual *Category* (Time Zone, Calendar Date, Duration, DST, Timestamps, String Representation, Other), a *Bug Pattern* (eleven categories such as `Using Naïve Datetime Incorrectly`, `Dropping Time Zone Information`, `Unspecified Fold`, etc.), and an *Erroneous Computation* dimension. The classification below is **rule-based on the (Category, Pattern, Computation) triple** rather than a per-bug deep dive: each of the 66 distinct triples appearing in the dataset is mapped to a Tempo architectural assessment, then applied to all 151 rows.

This means the rationale for two bugs with the same triple is identical, and the assessment is at the level of *bug class* not *individual root cause*. For a publication-grade claim, the ambiguous cases (mostly `General` and `Partial` rows) would warrant per-bug verification against the linked GitHub issue.

Source dataset: <https://github.com/cmu-pasta/date-time> (`analysis/data/bugs_analysis_base.tsv`).

## Architectural analysis by category

### Time Zone (74 bugs — 49% of the corpus, the dominant category)

This is the category Tempo's design most directly addresses. Tiwari et al.'s dominant sub-pattern is `Using Naïve Datetime Incorrectly` — mixing Python's naïve and aware datetimes, with the naïve side silently treated as local time in arithmetic or comparison. Tempo has **no naïve/aware distinction**: a `%Tempo{}` value with time-of-day either carries `:extended.zone_id` or explicitly does not, and no comparison or arithmetic path silently promotes one to the other. The runner-up sub-patterns (`Dropping Time Zone Information`, `Comparing Time Zones`) are addressed by `align/3` (zone preservation through set operations) and `Tempo.Compare.compare_endpoints/2` (zone-aware projection to gregorian UTC seconds). DST fall-back ambiguity (`Unspecified Fold`) is disambiguated through IXDTF offset notation per RFC 9557 §4.5 — `01:30:00-04:00[America/New_York]` vs `01:30:00-05:00[America/New_York]` picks the EDT or EST period unambiguously, and the disambiguator round-trips through string serialisation.

### String Representation (19 bugs)

Mostly `Incorrect Format Assumptions` and `Precision of Representation` bugs in parse/format paths. Tempo's parser implements ISO 8601-2:2019 and IXDTF draft-09, handling `Z`, named zones, offsets including `+10:00` and beyond, and preserves declared resolution on round-trip. The remaining bugs in this category are sub-second precision issues (a Tempo gap — see Duration / Timestamps below) and OOB range cases that depend on calendar implementation rather than format.

### Other (22 bugs — almost all Python 3.12 deprecation warnings)

21 of the 22 bugs in this category are the Python 3.12 deprecation of `datetime.utcnow()` / `datetime.utcfromtimestamp()`. Out of scope: Tempo exposes no global-state `utcnow()` equivalent, so the bug class can't arise.

### Calendar Date (11 bugs)

Mix of arithmetic bugs (e.g. 'last day of the month is wrong') and jurisdiction-specific holiday-calculation bugs in third-party libraries (`holidays` package: US Mississippi Confederate Memorial Day, etc.). Arithmetic bugs are addressed by Tempo's half-open materialisation — `~o"2026-02"` is the interval `[2026-02-01, 2026-03-01)` and the 'last day of February' is unambiguously the day before the exclusive upper bound. Holiday-table bugs are out of scope: Tempo provides multi-calendar arithmetic but not holiday tables.

### Duration (11 bugs)

Mostly `Unintuitive Arithmetic` bugs around Python's `relativedelta` (calendar-relative deltas). `Tempo.Duration` follows the same calendar-relative model (`P1M` is one calendar month, variable in days), so the conceptual subtleties remain — partial coverage. Sub-second duration precision, previously a gap, is now modelled (`:microsecond {value, precision}`); `PT0.5S` round-trips and arithmetic carries across the second.

### DST (6 bugs)

Tempo's DST handling is comprehensive, drawing on three mechanisms: (1) DST gaps (spring-forward, the non-existent hour) are *rejected* at parse time via `Tempo.ZoneGapError` — stronger than addressing, the bad value cannot be constructed; (2) DST folds (fall-back, the doubled hour) are disambiguated by IXDTF offset notation per RFC 9557 §4.5 — `01:30:00-04:00[America/New_York]` picks EDT, `01:30:00-05:00[America/New_York]` picks EST, and the disambiguator round-trips through string serialisation, more principled than Python's `fold` parameter which is a meta-attribute invisible to the string form; (3) enumeration across a DST transition emits the doubled hour twice with distinct shifts and skips the non-existent hour. All `Unspecified Fold` and DST-arithmetic patterns are addressed.

### Timestamps (8 bugs)

Bugs around Unix timestamps (`time_t`), pre-1970 representations, and tick generation. Tempo's gregorian-seconds projection (Erlang `:calendar.datetime_to_gregorian_seconds/1`) is rooted at year 0 and has no POSIX epoch limit, so several of these bug classes don't arise. Sub-second precision, previously a gap, is now modelled to microsecond (`:microsecond {value, precision}`, matching Elixir's cap).

## Reading the table

* **In scope · Avoided** means the bug class cannot arise in Tempo — the underlying API affordance that produces the bug in Python doesn't exist in Tempo's surface.

* **In scope · Addressed** means Tempo has explicit machinery for the case and the bug is prevented when the standard API is used.

* **In scope · Partial** means Tempo's design helps but doesn't eliminate the class — typically arithmetic with calendar-relative subtleties that remain by design.

* **In scope · Gap** means Tempo's design doesn't address this case. After sub-second (`:microsecond`) resolution was added there are **no gaps** against this corpus — the six former gaps (all sub-second precision in Duration and Timestamps) are now Addressed.

* **Out of scope** means the bug is a Python-language deprecation or a third-party-library-specific concern (holiday tables, etc.) that Tempo's architecture does not aim to replace.
