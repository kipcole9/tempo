## Details

### 1. Mask filling on year crashes

**Reproducer**

```elixir
iex> Enum.take(Tempo.from_iso8601!("156X"), 3)
** (Protocol.UndefinedError) protocol Enumerable not implemented for Integer…
```

**Root cause**

`Tempo.Mask.fill_unspecified(:year, :any, _, _)` returns `[Date.utc_today().year]` — a list containing a single integer. The mask path destructures this as `[target_range] = […]` and then does `Enum.reduce(target_range, …)`, iterating over what it assumes is a Range. Because `target_range` is bound to a bare integer, `Enum.reduce` hits `Protocol.UndefinedError` for `Integer`.

**Proposed fix**

Compute year-mask bounds directly from the mask rather than routing through the `:any` path. For mask `[1, 5, 6, :X]` the bounds are `1560..1569`. Works for positive and negative masks alike and avoids the current-year dependency for masked years.

### 2. Full second-precision values crash

**Reproducer**

```elixir
iex> Enum.take(Tempo.from_iso8601!("2022-06-15T10:30:00"), 3)
** (MatchError) no match of right hand side value: nil
```

**Root cause**

`Tempo.Enumeration.add_implicit_enumeration/1` destructures `Unit.implicit_enumerator(unit, calendar)` as `{unit, range}`. `Unit.implicit_enumerator` returns the entry from `@unit_after` — but `@unit_after` has no `:second` key, so the call returns `nil`. The `{unit, range} = nil` match fails.

**Proposed fix**

Guard the destructure and raise a clear `ArgumentError` when no implicit enumerator exists. Message explains the reason: "Cannot enumerate a Tempo at `:second` resolution — no finer unit is defined."

The alternative — adding `second: {:subsecond, 0..0}` to `@unit_after` — would invent a non-standard unit. Raising is the more honest choice; callers who want to iterate sub-seconds can build a `%Tempo.Duration{}` explicitly.

### 3. Metadata propagation

Blocked by gaps 1 and 2. The conformance harness's tests for IXDTF and qualification propagation all use second-resolution or masked inputs, both of which currently crash. Once 1 and 2 are fixed the harness should pass these cases — the existing `%{tempo | time: …}` struct update in `Enumeration.collect/1` preserves all unchanged fields.

### 4. Group expansion crashes

**Reproducer**

```elixir
iex> Enum.take(Tempo.from_iso8601!("2022Y5G2MU"), 3)
** (FunctionClauseError) no function clause matching in Tempo.Enumeration.do_next/3
```

Group syntax resolves to a token shape `do_next/3` doesn't recognise. This is a pre-existing bug outside the Enumerable-review plan's scope. Filed for separate follow-up.

### 5. Selection produces malformed time

**Reproducer**

```elixir
iex> Enum.take(Tempo.from_iso8601!("2022YL1MN"), 3)
# Yields a Tempo with time: [year: 2022, selection: {:month, 1}]
# The inner {:month, 1} tuple isn't a list — breaks downstream enumeration.
```

Selection tokens aren't unpacked by enumeration. Pre-existing; outside plan scope.

### 6. `Enumerable.Tempo.Interval` is not implemented

`lib/protocol/enumeration/range.ex` is empty. `Enum.take(interval, n)` raises `Protocol.UndefinedError` for every interval value. This is the big Step 5 of the plan (3–5 days).

### 7–8. Long-year shapes

Predicted by the plan. The harness confirms:

* `Y171010000S3` (significant-digits tuple) crashes with `no function clause matching in Tempo.Enumeration.do_next/3`.
* `Y17E8` enumerates in principle but over 1.7 × 10⁹ years is not practical.

Step 4 of the plan: detect these shapes and either compute a digit-bounded range or refuse with a clear error.

### 9. Component qualification semantics

Verified reachable once gap 1 is fixed (the mask path is independent of qualification). Plan recommends Option 1 (propagate verbatim via the natural struct update in `collect/1`) — no code change needed, only a regression test.

## Fixes landed in this session

* **Gap 1** (year mask) — `Tempo.Mask.fill_unspecified/4` now computes year-mask bounds directly (`[1, 5, 6, :X]` → `1560..1569`), filters candidates, and no longer depends on the misbehaving `:year, :any` branch. Mirror rewrite for the negative-year case. Positive / negative / fully-unspecified (`156X`, `-1XXX`, `XXXX`) all enumerate.

* **Gap 2b** (year+month+day with day-list) — `Tempo.Validation.resolve/2`'s year-month-day clause's guard was `is_number(day) or is_struct(day, Range)`. Extended to also accept `is_list(day)` so the `1985-XX-XX` / `1985-12-XX` / `-1XXX-XX-XX` paths conform the relative `[1..-1//1]` day range against days-in-month.

* **Gap 2** (second-precision crash) — `Tempo.Enumeration.add_implicit_enumeration/1` now raises a clear `ArgumentError` when `Unit.implicit_enumerator/2` returns `nil`. Instead of `** (MatchError) no match of right hand side value: nil`, callers see `"Cannot enumerate a Tempo at :second resolution — no finer unit is defined."`.

* **Gap 3** (metadata propagation) — verified via the harness. `%Tempo{}` struct updates in `Enumeration.collect/1` preserve `:extended`, `:qualification`, and `:qualifications` without any code change. Four regression tests pin this down.

## Fixes landed in the follow-up session

* **Gap 4** (group `2022Y5G2MU`) — added a `do_next/3` clause that unwraps `{:group, %Range{}}` into the standard range-iteration path (`[{unit, [range]} | t]`). The rest of the cycle machinery handles it transparently.

* **Gap 5** (selection `2022YL1MN`) — moved the `{:selection, _}` clause in `do_next/3` ABOVE the generic `{unit, value} when is_unit(unit, value)` clause. The previous ordering let `is_unit` match the selection's inner keyword list and destructively iterate it. Updated `explicitly_enumerable?/1` so a bare selection alone does not mark a Tempo as enumerable (it is a constraint, not a sequence).

* **Gap 7** (significant-digits enumeration) — added `do_next/3` clause for `{unit, {value, [significant_digits: n]}}` that expands the tuple into a digit-prefix range (`1950S2` → `[1900..1999]`). Guards against blocks larger than `@significant_digits_limit` (10 000 candidates) by raising a clear `ArgumentError`. Negative values enumerate in most-negative-first order. The helper `significant_digits_range/2` handles the digit arithmetic.

* **Gap 2c** (year-range + month-resolve) — extended `Tempo.Validation.resolve/2`'s `{:year, year}, {:month, months}` clause guard from `is_list(months) or is_integer(months)` to also accept `is_struct(months, Range)`. This lets the implicit month enumerator (`1..-1//-1`) conform against `months_in_year` when the year is itself a range (as it is during significant-digits iteration).

* **Gap 6** (`Enumerable.Tempo.Interval`) — implemented. `lib/protocol/enumeration/range.ex` is no longer empty. Closed intervals and open-upper intervals iterate forward one resolution-unit at a time from the `:from` endpoint; fully-open and open-lower intervals raise `ArgumentError` with a clear message. Iteration honours the half-open `[from, to)` convention — the upper bound is exclusive, matching `CLAUDE.md`'s architectural rule. `count/1`, `member?/2`, and `slice/1` return `{:error, __MODULE__}` for now; precise implementations are tracked with the set-operations milestone.

  Increment is calendar-aware: `increment_time/3` uses `calendar.months_in_year/1` and `calendar.days_in_month/2` for carry. All Keyword updates use `Keyword.replace!/3` (preserves position) rather than `Keyword.put/3` (removes + prepends) because the keyword-list order is the invariant `compare_time/2`, `inspect`, and `to_iso8601` all depend on.

## Still deferred

None of the original gaps remain. The harness assertions are now tightened to exact expected outputs rather than `{:ok, _} or {:crash, _}` matches, so any future regression surfaces as a clear test failure.

Future enhancements tracked elsewhere:

* Precise `count/1` and `member?/2` for `Tempo.Interval` — needs Tempo-to-Tempo comparison and wall-clock day arithmetic. Tracked with the set-operations milestone.

* Increment support for additional resolution units (`:week`, `:day_of_year`, `:day_of_week`, fractional units). The current `increment_time/3` covers year / month / day / hour / minute / second — adequate for every standard ISO 8601 interval shape and every harness test, but future EDTF shapes may want more.
